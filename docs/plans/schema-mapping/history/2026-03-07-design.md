# MySQL Database 到 PostgreSQL Schema 映射设计

## 1. 背景

当前 AProxy 的后端 PostgreSQL 连接在启动时通过配置固定到单个 PostgreSQL database。MySQL 客户端连接到 AProxy 时，可以在 DSN 中指定 database，也可以在连接后执行 `USE db_name` 切换逻辑库。

从当前实现看，AProxy 已经具备一部分“将 MySQL database 解释为 PostgreSQL schema”的行为：

- 后端 PostgreSQL database 在启动时固定，连接池按 `cfg.Postgres.Database` 建连。
- `COM_INIT_DB` 和 `USE db` 已经被处理。
- 当前处理方式是直接执行 `SET search_path TO <dbName>`。
- `SHOW TABLES`、`SHOW COLUMNS`、部分 schema cache 查询依赖 `current_schema()`。

这说明当前实现的真实语义并不是“动态切换 PostgreSQL database”，而是“在固定 PostgreSQL database 中切换 schema”。

但当前行为仍然不完整，存在以下问题：

- 连接串中的初始 MySQL database 未被统一映射到 PostgreSQL schema。
- `database` 和 `schema` 在会话模型、日志、缓存键中被混用，语义不清晰。
- `SET search_path TO <dbName>` 通过字符串拼接实现，存在安全和兼容性风险。
- 默认 schema 行为未被显式定义，导致对象可能意外落在 `public`。

本文档定义一套明确、稳定且可测试的映射方案：

在单个固定的 PostgreSQL database 内，将一个 MySQL database 映射为一个 PostgreSQL schema。

## 2. 目标与非目标

### 2.1 目标

1. 固定连接单个 PostgreSQL database，不因 MySQL 客户端 database 名称而切换 PostgreSQL database。
2. 将 MySQL 客户端指定的 database 名称自动映射为当前会话的 PostgreSQL schema。
3. 让连接串初始 database、`COM_INIT_DB`、`USE db` 三种入口的行为保持一致。
4. 让未带 schema 的 SQL、元数据查询、schema cache、DDL/DML 行为都基于同一套 schema 语义运行。
5. 明确默认 schema 策略、错误处理策略和安全约束。

### 2.2 非目标

1. 不支持在同一后端 PostgreSQL 连接中切换物理 database。
2. 不实现“一个 MySQL database 对应一个 PostgreSQL database”的映射模型。
3. 不在本次设计中引入跨 PostgreSQL database 的透明路由。
4. 不在本次设计中自动创建 schema，除非显式开启配置。

## 3. 设计结论

### 3.1 统一语义

AProxy 采用如下统一语义：

- PostgreSQL `database` 由 AProxy 启动配置固定决定。
- MySQL `database` 映射为 PostgreSQL `schema`。
- MySQL 客户端连接串中的 `/db_name` 与 `USE db_name` 的语义等价。
- 会话切换 database 的本质实现为切换 PostgreSQL `search_path`。
- 未指定 database 时，使用 AProxy 配置中的默认 schema。

### 3.2 对客户端的可见行为

对 MySQL 客户端而言，仍然表现为“连接到某个 database 并在其内执行 SQL”。

对 PostgreSQL 后端而言，所有 SQL 都在同一个物理 database 内执行，仅通过 `search_path` 或显式 schema 解析对象名。

### 3.3 推荐的 search_path 规则

推荐在切换逻辑库时设置：

```sql
SET search_path TO <mapped_schema>, public;
```

原因：

- 使未带 schema 的业务表优先解析到当前逻辑 schema。
- 保留对 `public` 中共享对象的兜底访问能力。
- 与 PostgreSQL 常见部署方式兼容。

如果业务要求严格隔离，也可以提供选项关闭 `public` fallback，改为：

```sql
SET search_path TO <mapped_schema>;
```

## 4. 当前实现分析

### 4.1 已有能力

当前代码已经具备以下基础：

1. 后端固定连接 PostgreSQL database。
2. `COM_INIT_DB` 已经由协议层接收并进入 `UseDB` 逻辑。
3. `USE db` 已通过 `SET search_path TO db` 实现 schema 切换。
4. 多个元数据模拟查询使用 `current_schema()` 获取当前 schema。
5. session-affinity 连接模型适合维持会话级 schema 状态。

### 4.2 现存问题

1. 初始连接 database 未确保在第一次获取 PG 连接后自动应用。
2. 会话中的 `Database` 字段实际承担的是“当前 schema”的语义，命名不准确。
3. schema cache、日志和错误信息延续了“database”命名，增加理解成本。
4. `SET search_path TO <dbName>` 通过字符串拼接执行，存在 SQL 注入和非法标识符问题。
5. 默认 schema 策略未显式配置，可能导致对象落在 PostgreSQL 默认 `public` schema。

## 5. 方案设计

### 5.1 会话模型调整

当前会话状态需要拆分“后端物理 database”和“当前逻辑 schema”。

建议将会话模型演进为：

```go
type Session struct {
    ID              string
    User            string
    BackendDatabase string
    CurrentSchema   string
    Charset         string
    Autocommit      bool
    InTransaction   bool
    LastInsertID    uint64
    ...
}
```

设计原则：

- `BackendDatabase` 只读，来源于代理配置。
- `CurrentSchema` 可变，来源于客户端初始 database、`COM_INIT_DB` 或 `USE db`。
- 所有当前使用 `session.Database` 作为 schema 的逻辑，逐步迁移为 `session.CurrentSchema`。

### 5.2 映射规则

默认映射规则：

```text
mysql_database_name -> postgres_schema_name
```

建议支持配置化覆盖：

```yaml
database_mapping:
  mode: schema
  default_schema: public
  fallback_to_public: true
  create_schema_if_not_exists: false
  rules:
    test: test
    analytics: ods_analytics
```

规则说明：

- `mode`: 固定为 `schema`。
- `default_schema`: 客户端未指定 database 时使用的 schema。
- `fallback_to_public`: 是否将 `public` 附加到 `search_path`。
- `create_schema_if_not_exists`: 是否允许自动创建 schema，默认关闭。
- `rules`: 允许 MySQL database 名与 PostgreSQL schema 名不一致时自定义映射。

### 5.3 统一入口

所有 schema 切换必须收敛到统一入口，不再在多个位置直接拼接 SQL。

建议新增两个核心方法：

```go
ResolveSchema(mysqlDB string) (string, error)
ApplySchema(ctx context.Context, conn *pgx.Conn, schema string) error
```

职责划分：

- `ResolveSchema` 负责根据配置规则将 MySQL database 名解析为 PostgreSQL schema 名。
- `ApplySchema` 负责安全地将 schema 应用到当前 PG 会话。

### 5.4 初始连接行为

客户端连接 AProxy 时，如果在握手阶段提供了初始 database，则应：

1. 将该值写入 `session.CurrentSchema`。
2. 第一次创建后端 PG 连接后，立即应用 schema。

这样可以保证以下两种写法行为一致：

```text
mysql://user:pass@host:3306/test
```

和：

```sql
USE test;
```

### 5.5 运行时切换行为

当收到 `COM_INIT_DB` 或 `USE db_name` 时：

1. 解析并校验目标 database 名。
2. 通过映射规则得到目标 schema。
3. 如果当前在事务中，按策略决定是否允许切换。
4. 更新 `session.CurrentSchema`。
5. 对当前 PG 连接执行 `SET search_path`。

推荐策略：

- 默认禁止在事务中切换 schema。
- 如果客户端在事务中执行 `USE db`，返回 MySQL 风格错误。

原因：

- MySQL 中切换 database 通常不期望改变事务对象解析范围。
- 在事务中切换 schema 会使当前事务的对象解析行为变得不透明。

### 5.6 默认 schema 行为

如果客户端未指定 database：

1. 使用 `database_mapping.default_schema`。
2. 在第一次 PG 连接可用时执行 `SET search_path`。

这可以消除“未指定时默认落到 `public`”的隐式行为，将其变成明确配置。

### 5.7 元数据兼容策略

所有 MySQL 元数据命令的默认解析应与 `CurrentSchema` 保持一致。

具体包括：

1. `SHOW TABLES` 默认查询当前 schema。
2. `SHOW COLUMNS` / `DESCRIBE` 默认解析当前 schema 下的目标表。
3. `SHOW DATABASES` 返回受控的逻辑 database 列表；这些逻辑名可映射到 PostgreSQL schema，但不等同于 PostgreSQL 物理 database 列表。
4. schema cache 的键应按 `schema.table` 组织，而不是继续使用模糊的 `database.table` 表示。

### 5.8 DDL / DML 行为

在该方案下：

- `CREATE TABLE users (...)` 实际创建为 `CurrentSchema.users`。
- `SELECT * FROM users` 优先解析到 `CurrentSchema.users`。
- `INSERT INTO users ...`、`UPDATE users ...`、`DELETE FROM users ...` 都在当前 schema 下执行。
- schema cache 与 DDL 失效逻辑均基于 `CurrentSchema` 运行。

## 6. 安全设计

### 6.1 不允许直接拼接 schema 标识符

当前直接执行：

```sql
SET search_path TO <dbName>
```

如果 `<dbName>` 直接来自客户端输入，会产生两个问题：

1. 恶意输入可能构造注入语句。
2. 合法但包含特殊字符或大小写的 schema 名可能被 PostgreSQL 错误解析。

### 6.2 建议安全策略

推荐采用“双保险”：

1. 对 schema 名做严格格式校验。
2. 通过安全 quoting 应用到 `search_path`。

推荐允许的 schema 名模式：

```text
^[A-Za-z_][A-Za-z0-9_]*$
```

如果未来必须支持更复杂的 schema 名，再引入安全的 PostgreSQL identifier quoting 工具，但默认不放开。

### 6.3 schema 存在性校验

建议在切换 schema 时增加存在性校验：

1. 若 schema 不存在，直接返回明确错误。
2. 若 `create_schema_if_not_exists` 为 true，则先创建再切换。
3. 默认不自动创建，以避免拼写错误产生脏 schema。

## 7. 错误处理策略

### 7.1 schema 不存在

当客户端指定的 MySQL database 无法映射到 PostgreSQL schema，或 schema 不存在时：

- 返回 MySQL 风格错误。
- 错误信息中明确说明目标逻辑库不存在或不可访问。

### 7.2 非法 database 名称

当客户端提供非法名称时：

- 拒绝切换。
- 返回语义清晰的错误，而不是让 PostgreSQL 原始语法错误直接暴露给客户端。

### 7.3 事务中切换 schema

默认返回错误，避免行为不确定。

## 8. 配置设计

建议在现有配置中新增映射配置段：

```yaml
postgres:
  host: "localhost"
  port: 5432
  database: "app"
  user: "postgres"
  password: ""
  max_pool_size: 200
  connection_mode: "session_affinity"
  ssl_mode: "disable"

database_mapping:
  mode: "schema"
  default_schema: "public"
  fallback_to_public: true
  create_schema_if_not_exists: false
  validate_schema_name: true
  rules:
    test: "test"
```

字段说明：

- `postgres.database`: 固定后端物理 database。
- `database_mapping.mode`: 当前固定为 `schema`。
- `default_schema`: 默认逻辑库对应 schema。
- `fallback_to_public`: 是否把 `public` 放入 `search_path`。
- `create_schema_if_not_exists`: 缺失 schema 时是否自动创建。
- `validate_schema_name`: 是否启用 schema 名格式校验。
- `rules`: 自定义映射规则。

## 9. 关键流程

### 9.1 首次连接

```text
MySQL client connects
  -> 握手阶段得到 initial database
  -> 创建 Session
  -> 解析 initial database 为 CurrentSchema
  -> 首次获取 PG 连接
  -> ApplySchema(CurrentSchema)
  -> 后续 SQL 在该 schema 下执行
```

### 9.2 运行时 USE

```text
Client sends USE db_name
  -> ResolveSchema(db_name)
  -> 校验 schema 名和存在性
  -> 若在事务中则按策略拒绝
  -> 更新 Session.CurrentSchema
  -> ApplySchema(CurrentSchema)
  -> 返回 OK
```

### 9.3 未指定 database

```text
Client connects without database
  -> Session.CurrentSchema = default_schema
  -> 首次获取 PG 连接时应用 default_schema
```

## 10. 对现有模块的影响

### 10.1 `pkg/session`

- `Database` 字段需要重新命名或拆分。
- 新增 `CurrentSchema` 访问方法。
- 与 schema cache、事务、元数据查询的交互要切换到 schema 语义。

### 10.2 `pkg/protocol/mysql/handler`

- `UseDB` 逻辑改为统一调用 schema 解析与应用入口。
- 第一次获取 PG 连接后要自动应用 `session.CurrentSchema`。
- DDL 失效缓存逻辑要使用 schema 维度。

### 10.3 `pkg/mapper/show`

- `HandleUseCommand` 不再自行拼接 `SET search_path`。
- `SHOW DATABASES` 的返回语义要在文档和注释中明确为 schema 列表。
- `SHOW TABLES`、`SHOW COLUMNS` 等逻辑继续基于 `current_schema()`，但变量命名要统一为 schema。

### 10.4 `internal/config`

- 新增 `DatabaseMappingConfig`。
- 增加配置默认值和校验逻辑。

### 10.5 `pkg/schema`

- cache key 命名从模糊的 `database.table` 语义明确为 `schema.table`。
- 相关注释和日志同步更新。

## 11. 测试策略

### 11.1 单元测试

1. `ResolveSchema` 正常映射。
2. `ResolveSchema` 自定义规则映射。
3. 非法 schema 名被拒绝。
4. 默认 schema 逻辑正确。
5. `fallback_to_public` 生成的 `search_path` 正确。
6. 事务中切换 schema 的行为正确。

### 11.2 集成测试

1. 使用 DSN 指定 `/test` 连接，建表后对象落在 `test` schema。
2. 连接后执行 `USE analytics`，后续建表和查询落在 `analytics` schema。
3. 未指定 database 时，建表落在默认 schema。
4. `SHOW TABLES` 仅返回当前 schema 的表。
5. `SHOW COLUMNS` / `DESCRIBE` 能正确解析当前 schema 的表结构。
6. schema 不存在时返回预期错误。
7. 非法 database 名返回预期错误。

### 11.3 回归测试

1. 现有 `SHOW DATABASES`、`SHOW TABLES`、`USE database` 测试需要更新语义描述。
2. AUTO_INCREMENT 检测和 schema cache 不能因字段改名而回归。
3. 事务、prepared statement、CDC、binlog 相关路径不能受 schema 切换影响。

## 12. 迁移与兼容性

### 12.1 向后兼容

当前已有 `USE db -> SET search_path TO db` 的基础行为，因此该设计主要是“显式化和完整化”，不是彻底重构语义。

### 12.2 需要同步更新的文档

1. README 中关于 `COM_INIT_DB`、`USE database`、`SHOW DATABASES` 的描述。
2. 配置示例中关于 `postgres.database` 的说明。
3. RUNBOOK 中关于排查“表建到哪里”的说明。
4. 测试文档中对“database”和“schema”的术语澄清。

## 13. 风险与取舍

### 13.1 优点

1. 与 PostgreSQL 连接模型一致，不需要动态切物理 database。
2. 与现有 session-affinity 设计兼容。
3. 对 MySQL 客户端保持较高透明度。
4. 便于通过 `current_schema()` 实现 SHOW/元数据兼容。

### 13.2 风险

1. 用户可能误以为 `SHOW DATABASES` 是 PG 物理 database 列表。
2. 若继续混用 database/schema 术语，会导致排障困难。
3. schema 切换如果未统一入口，容易出现会话状态和 PG 状态不一致。

### 13.3 取舍结论

在“固定单个 PostgreSQL database”的约束下，将 MySQL database 映射为 PostgreSQL schema 是最合理、最稳定、最符合当前代码基础的方案。

## 14. 实施建议

建议按以下顺序落地：

1. 引入 `DatabaseMappingConfig` 与默认配置。
2. 在 session 中拆分 `BackendDatabase` 和 `CurrentSchema`。
3. 实现统一的 `ResolveSchema` / `ApplySchema` 入口。
4. 在握手初始库、`COM_INIT_DB`、`USE db` 三个入口统一接入。
5. 修复 `SET search_path` 的安全问题。
6. 调整 SHOW、schema cache、日志、错误信息中的术语和键名。
7. 补齐单元测试和集成测试。

## 15. 最终结论

本设计将 AProxy 的“固定后端 PostgreSQL database”与“客户端可切换逻辑库”的需求统一起来：

- 后端始终连接单个 PostgreSQL database。
- MySQL database 作为逻辑库概念映射到 PostgreSQL schema。
- 通过会话级 `search_path` 保持对象解析一致性。
- 通过统一的映射、校验和配置机制保证安全性、可预测性和可测试性。

这是当前架构下最合适的工程实现方案。
